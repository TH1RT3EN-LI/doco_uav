UAV Control Stack
=================

Stage 3 Control Architecture
----------------------------

The active control path is now PX4-native:

- OpenVINS raw odometry:
  `/ov_msckf/odomimu`

- PX4 external-vision input:
  `/fmu/in/vehicle_visual_odometry`
  Produced by `openvins_px4_vision_bridge_node`.
  The bridge no longer applies EV position translation compensation.
  PX4-side lever-arm compensation should be configured with `EKF2_EV_POS_*`.
  The bridge still converts ENU/FLU attitude into PX4 NED/FRD attitude.

- PX4 fused local state output:
  `/uav/state/odometry_px4`
  Produced by `uav_state_bridge_node`.
  This is the authoritative control state for Jetson-side logic.

- Jetson command inputs:
  `/uav/control/position_delta`
  `/uav/control/position_yaw`
  `/uav/control/position_keep_yaw`
  `/uav/control/setpoint/velocity_body`

- PX4 offboard outputs from Jetson:
  `/fmu/in/offboard_control_mode`
  `/fmu/in/trajectory_setpoint`
  `/fmu/in/vehicle_command`

Control Roles
-------------

- PX4 owns the inner control loops.
  Position, velocity, attitude, and landing execution stay inside PX4.

- Jetson owns task logic and high-level targets only.
  Jetson reads `/uav/state/odometry_px4`, then sends either:
  position setpoints through `/uav/control/position_yaw`, or
  body velocity setpoints through `/uav/control/setpoint/velocity_body`.

- OpenVINS no longer closes the outer loop inside `uav_control_node`.
  The old OV pose-error controller and OV fallback landing path were removed.

Public Service Surface
----------------------

- `/uav/control/command/takeoff`
  Capture the current PX4 fused local pose and raise the local `z` target
  by `takeoff_height_m`.
  `uav_control_node` then publishes PX4 position setpoints until the target
  is reached, after which it transitions to hold.

- `/uav/control/command/hold`
  Capture the current PX4 fused local pose and keep publishing that pose
  as a PX4 position setpoint.

- `/uav/control/command/position_delta`
  Body-relative position-step request.
  Input is `uav_bridge/srv/BodyPositionDelta` with fixed body FLU semantics:
  `x` forward, `y` left, `z` up.
  `position_delta_node` snapshots the current PX4 fused pose, rotates the
  body-relative delta into local ENU using the current fused yaw, then publishes
  the resulting absolute target through `/uav/control/position_keep_yaw`.

- `/uav/control/position_yaw`
  Direct PX4 local position plus yaw target.
  Default frame is `uav_odom`.
  The target is interpreted in the same local ENU frame published by
  `/uav/state/odometry_px4`, then converted to PX4 local NED.
  Orientation is required; commands without a valid quaternion are rejected.
  Only yaw is consumed from the quaternion; roll and pitch are ignored by the
  PX4 position-setpoint path.

- `/uav/control/position_delta`
  Relative local-position target helper.
  Input is `geometry_msgs/Vector3`.
  Semantics are fixed to body FLU:
  `x` forward, `y` left, `z` up.
  `position_delta_node` rotates the delta into local ENU using the current
  fused yaw, then converts it into an absolute
  `/uav/control/position_keep_yaw` target.

- `/uav/control/position_keep_yaw`
  Direct PX4 local position target while keeping the current fused yaw.
  Input is `geometry_msgs/PointStamped` in the same local ENU frame as
  `/uav/state/odometry_px4`.
  `uav_control_node` snapshots the current PX4 fused yaw when the command is
  received and uses that frozen yaw for the generated PX4 setpoint.
  A fresh control state is required or the command is rejected.

- `/uav/control/setpoint/velocity_body`
  Direct PX4 velocity offboard target.
  Input is body-frame FLU velocity plus ENU yaw-rate command.
  `uav_control_node` uses the current PX4 fused attitude to rotate the body
  velocity into world ENU, then converts to PX4 NED velocity.

- `/uav/control/command/land`
  Pure PX4 land request.
  No Jetson-side outer-loop landing fallback remains.

- `/uav/control/command/position_mode`
  Request PX4 Position mode.

- `/uav/control/command/abort`
  Force PX4 Stabilized mode for operator takeover.

- `/uav/control/command/disarm`
  Send PX4 disarm command and clear any active offboard targets.

Launch Defaults
---------------

- `launch/minimal_control.launch.py`
  `control_state_topic=/uav/state/odometry_px4`
  `execution_state_topic=/uav/state/odometry_px4`
  `position_command_frame_id=uav_odom`
  `trajectory_input_odom_topic=/uav/state/odometry_px4`
  `px4_timestamp_source=px4_timesync`

- `launch/openvins_orbbec.launch.py`
  `start_px4_vision_bridge=true`
  PX4 EV bridge sensor translation defaults are `0.0`.
  Use PX4 `EKF2_EV_POS_*` for EV lever-arm compensation.

Important Runtime Semantics
---------------------------

- EKF local reset handling:
  `uav_control_node` listens to `/fmu/out/vehicle_local_position`.
  If PX4 resets local XY/Z/heading, stored hold/position targets are shifted
  by PX4 reset deltas so the commanded world target stays consistent.

- Timestamp handling:
  `uav_control_node`, `uav_state_bridge_node`, and `openvins_px4_vision_bridge_node` support
  `px4_timestamp_source=px4_timesync`.
  On hardware this should be the preferred default so EV input, bridged PX4
  state, and offboard setpoints use PX4-aligned timestamps.
  `uav_state_bridge_node` maps PX4 `timestamp_sample` into ROS time and holds
  publication until timesync is ready.

Removed Legacy Semantics
------------------------

- `uav_control_node` no longer treats `/uav/state/odometry` as the
  authoritative control frame.

- `openvins_state_bridge_node` and the OpenVINS monitor output
  `/uav/state/odometry` were removed from the hardware bringup path.
  Use `/ov_msckf/odomimu` directly if raw OpenVINS odometry is needed.

- OV PID outer-loop hold/takeoff/position control has been removed.

- Obsolete compatibility parameters and launch arguments were removed from the
  active stack, including `ov_hold_*`, `ov_target_*`, `ov_fault_*`, and
  `motion_guard_*`.

- Jetson is no longer the primary closed-loop position controller.
  Jetson now issues PX4-native setpoints and PX4 executes the closed loop.
