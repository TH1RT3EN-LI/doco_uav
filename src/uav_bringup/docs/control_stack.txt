UAV Control Stack
=================

Public Launch Entry Points
--------------------------

- `launch/minimal_control.launch.py`
  Base control chain.
  Starts `uav_control_node`, `rc_safety_mux_node`, PX4 state bridge,
  trajectory publisher, optional mono recorder, and the local camera
  helper launches they depend on.
  It does not launch OpenVINS or the OpenVINS state bridge anymore;
  start `launch/openvins_orbbec.launch.py` separately when OV control
  state is required.

- `launch/visual_landing.launch.py`
  Visual landing composition entry.
  Includes `minimal_control.launch.py`, then adds
  `height_measurement_bridge_node`, `aruco_detector_node`, and
  `visual_landing_node`.

- `launch/test_home_visual_landing.launch.py`
  Test-only entry.
  Includes `visual_landing.launch.py`, then starts
  `home_visual_landing_test_node.py`.

- `launch/openvins_orbbec.launch.py`
  Perception and bridging leaf launch.
  Hosts the Orbbec and OpenVINS related stack only; it is not a
  control-stack entry point.

Public Service Surface
----------------------

- Base control services:
  `/uav/control/command/takeoff`
  `/uav/control/command/hold`
  `/uav/control/command/position_mode`
  `/uav/control/command/land`
  `/uav/control/command/abort`
  `/uav/control/command/disarm`

- Visual landing task services:
  `/uav/visual_landing/command/start`
  `/uav/visual_landing/command/stop`

- Test-only services:
  `/uav/test/home_takeoff`
  `/uav/test/home_visual_land`

`/uav/control/command/abort` and `/uav/control/command/position_mode`
are intentionally both kept.

- `abort` is the emergency handover path used by RC safety to force PX4
  Position mode.
- `position_mode` is the normal explicit mode-switch interface for
  operators and scripts.

State Roles
-----------

- `/uav/state/odometry`
  Authoritative OV control state.
  `uav_control_node` uses this topic for `hold`, `takeoff`, and
  `/uav/control/pose`.

- `/uav/state/odometry_px4`
  PX4 execution and debug state.
  `uav_control_node` uses this topic for execution-side attitude and
  body velocity only, and `visual_landing_node` uses it as its default
  state input.

Control Semantics
-----------------

- `/uav/control/command/takeoff`
  Relative takeoff in OV coordinates.
  The node captures the current OV pose, raises the OV `z` target by
  `takeoff_height_m` (default `0.70 m`), and runs an OV outer-loop
  controller that outputs
  body velocity and yaw-rate commands for PX4 to execute.

- `/uav/control/command/hold`
  OV pose hold.
  The node captures the current OV pose and keeps that pose using the
  same OV outer-loop body-velocity controller.

- `/uav/control/pose`
  Absolute target in the OV/map control frame.
  Position and yaw error are computed in OV, converted into body
  velocity and yaw-rate commands, then sent to PX4 through offboard
  velocity setpoints.

- `/uav/control/command/land`
  PX4-native land path.
  Landing mode itself is still requested directly from PX4 instead of
  being reimplemented in OV coordinates.

- OV fault handling
  The control stack checks OV timeout, pose jump, and invalid values.
  If a fault occurs during OV-driven automatic control, it exits
  automatic control immediately and requests PX4 Position mode for
  manual takeover.

Current Defaults
----------------

- `motion_guard_enabled` defaults to `false` in the public control
  launches.
- OV outer-loop defaults are conservative for indoor bringup:
  `ov_hold_max_vxy=0.20 m/s`
  `ov_hold_max_vz=0.10 m/s`
  `ov_hold_max_yaw_rate=0.20 rad/s`
  `max_velocity_setpoint_mps=0.25 m/s`
  `max_acceleration_setpoint_mps2=0.35 m/s^2`
- RC safety remains:
  `CH5 HIGH -> /uav/control/command/abort`
  `CH5 LOW/MIDDLE -> no action`
  `CH7 -> PX4 kill path outside ROS services`

Removed Legacy Paths
--------------------

- `launch/minimal_visual_landing.launch.py` has been removed.
- `uav_bridge/src/offboard_bridge_node.cpp` and its legacy
  `/uav/control/*` services have been removed.
- The whole `uav_mode_supervisor` package and its `/uav/mode*`
  interface surface have been removed.
